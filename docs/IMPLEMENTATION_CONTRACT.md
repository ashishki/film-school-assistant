## 1. System Invariants (Must NOT Break)

### INV-1: Single-User Security Guard
The chat_id guard in src/bot.py MUST run before any handler, in handler group=-1000.
No message from a non-allowed chat_id may ever produce a response or trigger a DB write.

### INV-2: Voice Audio Never Leaves the Server
Audio files are processed locally only. Whisper runs on-device. OGG and WAV files are never transmitted to any external service. After transcription, temp WAV is deleted.

### INV-3: Confirmation Required Before Entity Save (Voice/NL)
Entities originating from voice or NL input MUST NOT be saved to the database until the user explicitly confirms (via /confirm, inline button, or confirmed word). parsed_events row is created immediately; entity row is created only after confirmation.

### INV-4: Explicit Commands Save Immediately
Commands (/note, /idea, /deadline, /homework) save entities immediately without a confirmation step. This asymmetry is intentional and must not be removed.

### INV-5: Reminder Idempotency
The reminder_log table UNIQUE(deadline_id, days_before) constraint enforces idempotency. send_reminders.py MUST check reminder_log before sending. This constraint must not be relaxed.

### INV-6: Weekly Summary Deduplication
send_summary.py MUST NOT send a summary if weekly_reports already has a record with sent_at set for the current week_start. Duplicate Telegram sends are not acceptable.

### INV-7: DB File is the System of Record
SQLite at DB_PATH is the only persistent store. In-memory state (src/state.py) is ephemeral and may be lost without notice. No feature may assume state survives a bot restart.

### INV-8: All Secrets via Environment Variables
Secrets (TELEGRAM_BOT_TOKEN, LLM_API_KEY, TELEGRAM_ALLOWED_CHAT_ID) MUST be loaded from environment variables via src/config.py. No secrets may be hardcoded in source files.

## 2. API / Behavior Guarantees

### G-1: Anthropic API Retry Policy
openclaw_client.py retries up to 3 times with exponential backoff (0.5s × attempt) on connection errors, timeouts, and 5xx responses. This policy applies to all LLM calls in the bot process.

### G-2: JSON Response Enforcement
All LLM calls that expect structured data MUST use complete_json() from openclaw_client.py. Raw text completion is not acceptable for extraction or review flows.

### G-3: Telegram Long-Polling
The bot uses long-polling (not webhooks). Application.run_polling() is the startup method. This must not be changed without also updating the systemd unit and deployment documentation.

### G-4: Whisper Model Caching
The Whisper model is loaded once and cached in transcriber.py module-level variable. It MUST NOT be reloaded per request. Reloading on each voice message would be a performance regression.

### G-5: Async DB Access
All database operations in the bot process MUST use async functions from src/db.py via aiosqlite. Synchronous SQLite calls (sqlite3 module) are only permitted in standalone scripts (send_reminders.py, send_summary.py, smoke_test_db.py).

### G-6: Cascade Constraint Respected
The schema uses FK constraints (PRAGMA foreign_keys=ON must be set at connection time). Deletes that would orphan child rows must use cascading deletes or explicitly clean up children first.

## 3. Failure Behavior

### F-1: LLM Extraction Failure
If Haiku extraction returns invalid JSON or fails after 3 retries: discard the pending entity, send user a message ("Could not parse your message. Please try the explicit command instead."). Do not leave state.pending_entity in a broken partial state.

### F-2: Whisper Transcription Failure
If Whisper fails: send user a message, delete temp WAV, do NOT insert a transcript row. The voice_inputs row is already inserted; processed_at remains NULL to indicate failure.

### F-3: Telegram API Failure in Scripts
If Telegram send fails in send_reminders.py or send_summary.py: log the error. Do NOT insert into reminder_log (so the next run will retry). Currently NOT implemented — see T-O1 in tasks.md.

### F-4: DB Connection Failure
If the SQLite DB cannot be opened at startup: bot.py MUST exit with a non-zero status code. systemd Restart=on-failure will retry. This is the correct behavior.

### F-5: Unknown Command
If an unknown command is received: bot does NOT respond (no "unknown command" message). This is current behavior and must be preserved to avoid spam in group contexts.

## 4. Tool-Use Rules (Profile: ON)

### TU-1: Model Selection Is Intentional
- `LLM_MODEL_INTENT` (default: Haiku) — fast, cheap, extraction only
- `LLM_MODEL_REVIEW` (default: Sonnet) — high quality, creative critique + summary

Do not swap models without explicit justification. Using Sonnet for extraction would be wasteful. Using Haiku for review/summary degrades output quality.

### TU-2: Prompt Versions Are Code
REVIEW_SYSTEM_PROMPT and EXTRACTION_SYSTEM_PROMPT are versioned in code. Changes to these prompts are behavioral changes and must be treated as code changes (commit, review, test).

### TU-3: External Audio Prohibition (see INV-2)
No new integration may transmit audio to an external API (e.g., Anthropic, OpenAI Whisper API). Local processing only.

## 5. Constraints from Current System

### C-1: Single-User Architecture
The current schema (single TELEGRAM_ALLOWED_CHAT_ID, no user_id on entities) does not support multiple users. Any multi-user feature requires a formal schema migration and auth redesign — not incremental patching.

### C-2: SQLite Concurrency
SQLite allows one writer at a time. The bot process (aiosqlite) and scheduled scripts (synchronous sqlite3) may write concurrently. WAL mode should be enabled (or verified) to prevent write conflicts. This is currently unverified — see open findings in CODEX_PROMPT.md.

### C-3: No Rollback on Multi-Step Flows
The voice confirmation flow inserts rows in multiple tables (voice_inputs, transcripts, parsed_events) before confirmation, then inserts the entity row on confirm. There is no transaction rollback if a step fails mid-flow. This is an accepted limitation for MVP.

### C-4: Systemd-Dependent Scheduling
Scheduled tasks depend on systemd. This system cannot run on Windows or environments without systemd. Any migration to a different scheduler (cron, APScheduler) requires updating all 8 systemd unit files and the deployment guide.
