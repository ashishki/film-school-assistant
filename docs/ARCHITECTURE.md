## 1. System Overview

**Film School Assistant** is a private, single-user Telegram bot designed as a structured creative workflow assistant for a film school student. It captures, organizes, and manages creative work through text and voice input, with automated reminders and weekly summaries.

**Core design principle:** Structured-first, conversational-second. Commands produce deterministic results. NL input is an enhancement layer, not the primary path. Data is stored in SQLite; the Telegram interface is a thin intake layer.

**Deployment:** Private VPS (OpenClaw HER instance), single user, no cloud storage dependencies.

## 3. Runtime Flow

### 3.1 Text Command (e.g., `/note`, `/deadline`)

```
Telegram message
  → chat_guard (pre-handler, group=-1000): drop if chat_id ≠ ALLOWED
  → CommandHandler dispatch
  → handler: parse args (regex/split), get_state(chat_id) for active_project_id
  → db.py: create_* CRUD call
  → reply_text confirmation to user
```
Entity is stored immediately; no confirmation step required.

### 3.2 Voice Input

```
Telegram voice message
  → voice_handler
  → Check: pending_entity in state? → reject if yes
  → db: insert voice_inputs row
  → voice.py: download OGG via get_file(), convert to WAV (ffmpeg 16kHz mono)
  → transcriber.py: Whisper inference (small model, local)
  → db: insert transcripts row
  → Show transcript to user
  → openclaw_client: Haiku call → extract {entity_type, content, project_hint, due_date}
  → db: insert parsed_events row (confirmed=0)
  → state: set pending_entity, pending_entity_type
  → Show confirmation prompt + inline buttons (/confirm, /discard, /edit)
```

### 3.3 Confirmation Flow (Voice & NL)

```
Pending entity exists in state
  → User sends: confirm word / /confirm / inline "Confirm" button
  → _do_confirm(): validate due_date, call create_*, confirm_parsed_event, clear_pending
     OR
  → User sends: discard word / /discard
  → clear_pending, reply "Discarded"
```

### 3.4 Free-Text Chat (Phase 5+)

```
Non-command text message, no pending entity
  → natural_confirm_handler (group=1): intercept confirm/discard if pending entity exists
  → chat_handler_wrapper (group=2): route to handle_chat()
      → check daily LLM limit (get_llm_calls_today)
      → build messages list from conversation_history + user message
      → AsyncAnthropic.messages.create with TOOLS (Haiku model)
      → if stop_reason == "tool_use": execute tools via execute_tool(), append tool results, loop
      → loop guard: MAX_TOOL_ROUNDS = 5; if exceeded, return last assistant message
      → update conversation_history via add_message()
      → reply with final text
```
Note: chat handler saves entities directly (no confirmation step). Voice path retains confirmation flow (3.3).

### 3.5 Scheduled Scripts (systemd timers)

| Script | Trigger | Action |
|---|---|---|
| send_reminders.py | Daily 08:00 | Query active deadlines, check reminder_log dedup, send Telegram for 7d/3d/1d/0d |
| send_summary.py | Monday 09:00 | 7-day snapshot → Sonnet narrative → weekly_reports → Telegram |
| cleanup_audio.py | Daily 03:00 | Delete .ogg/.wav older than AUDIO_RETENTION_DAYS (default 30) |
| backup_db.sh | Daily 02:00 | SQLite .backup + integrity check, keep 7 most recent |

## 6. Capability Profiles

| Profile | Status | Justification |
|---|---|---|
| **RAG** | **OFF** | No retrieval over a corpus. Data access is direct SQLite queries. No vector index, no embedding pipeline. |
| **Tool-Use** | **ON** | Anthropic API calls (Haiku, Sonnet), Whisper inference, Telegram Bot API, FFmpeg subprocess. LLM is called with structured prompts; tool-use is implicit in the architecture. |
| **Agentic** | **ON** | Multi-step tool-calling loop in chat_handler.py: Claude selects tools, executes them via execute_tool(), and loops until end_turn or MAX_TOOL_ROUNDS (5) guard fires. User-triggered but self-directing within each exchange. |
| **Planning** | **OFF** | No planning horizon or task decomposition inside the bot. Weekly summary synthesis is LLM text generation, not planning. |

## 8. Risks

| Risk | Severity | Notes |
|---|---|---|
| In-memory state lost on restart | Medium | Pending voice/NL entities are lost. User sees no error; entity simply never gets saved. |
| No rate limiting on Anthropic calls | Medium | Runaway costs possible if bot is flooded (low risk given single-user, but not zero) |
| No project creation via bot | Medium | User must seed projects via SQL or init script. Missing command blocks primary workflow. |
| Reminder timer failure is silent | Medium | If systemd timer fails, no alert is sent to user. No monitoring in place. |
| Summary LLM call not idempotent | Low | If summary.service is triggered twice in one day, costly LLM call is made twice even though report is already marked sent |
| Audio path not validated | Low | f-string path construction in voice.py; low risk in single-user context but worth noting |
| Mixed-language user-facing errors | Low | Some error strings are Russian, some English. Inconsistent UX. |
